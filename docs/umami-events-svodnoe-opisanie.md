# Umami Events — Сводное описание

Полный реестр событий аналитики, сгруппированный по страницам приложения.

---

## Глобальные / Сессионные

| Событие                            | Способ вызова                                                    | Описание                                                        |
| ---------------------------------- | ---------------------------------------------------------------- | --------------------------------------------------------------- |
| `trackUmamiSession` (signin)       | `window.umami.identify()` — программно после входа               | Идентификация пользователя при авторизации                      |
| `trackUmamiSession` (signup)       | `window.umami.identify()` — программно после регистрации         | Идентификация пользователя при регистрации                      |
| `trackUmamiSession` (session_init) | `window.umami.identify()` — программно при восстановлении сессии | Идентификация пользователя при инициализации сессии             |
| `role-change`                      | `window.umami.track()` — программно при смене роли               | Переключение роли (репетитор ↔ ученик), свойства: `from`, `to`  |
| `onboarding-complete`              | `window.umami.track()` — программно при завершении тура          | Завершение онбординг-тура, свойство: `layout` (tutor / student) |

## Product / Outcome events

Программные события продуктовой аналитики (`trackProductEvent` из `common.utils`). Отправляются только в Umami после успешного действия или при выполнении meaningful-условий. Не пишутся в основную БД.

> Существующие события остаются историческими UI/clickstream-событиями.
> Product/outcome events используются для расчёта активности, активации и retention.

---

### Воронка занятия и связи между событиями

```
[Репетитор]                                  [Студент]
     │                                             │
student_invited_success                   invite_accepted_success
     │                                             │
lesson_created_success                            ...
     │
     ▼ клик «Начать урок»
     │ → POST /access-tokens/ (tutor)
     │   ┌─ ошибка (не 409) → call_connection_failed { token_error }
     │   └─ 409 → reactivateCall → повторный POST
     │            ┌─ ошибка → call_connection_failed { token_error }
     │            └─ OK ──────────────────────────────────────────────────────────────┐
     │                                                                                │
lesson_started ◄──── (только при успехе)         lesson_joined ◄─── (студент, успех)│
     │                                                    │                           │
     └──────────────────── LiveKit room ─────────────────┘                           │
                                │                                                     │
                         call_connected (обе роли, однократно)                       │
                                │                                                     │
                 ┌──────────────┴──────────────┐                                     │
           браузер отказал                 соединение OK                             │
           в камере/микро                       │                                     │
                │                   setInterval 30s (только тьютор)                  │
       call_connection_failed            │                                            │
       { permission_error }     lesson_duration_reached                              │
                                  duration_min: 5 / 15 / 30 / 45                    │
                                  students_count: N                                  │
                                         │                                            │
                                  при disconnect (≥5 мин):                           │
                                    lesson_finished (обе роли)                       │
                                                                                     │
   board_opened ◄─────────────── открытие доски ──────────────────────────────────► │
   board_used_meaningfully ◄───── meaningful use ────────────────────────────────► │
```

**Ключевые связи:**

- `lesson_started` → ≤ 1 `call_connected` (тьютор) — разрыв означает, что получил токен, но не подключился
- `call_connected` ≥ `lesson_duration_reached` (duration_min=5) — разрыв означает короткие сессии < 5 мин
- `lesson_duration_reached` (45) ≤ (30) ≤ (15) ≤ (5) — монотонно убывает, иначе баг

### Что умеет и чего не умеет эта аналитика

| Вопрос                                             | Можно ответить? | Как                                                                       |
| -------------------------------------------------- | --------------- | ------------------------------------------------------------------------- |
| Сколько тьюторов провели урок за неделю?           | ✅              | `lesson_duration_reached`, `duration_min=15`, уникальные пользователи     |
| Сколько занятий длилось 45+ мин?                   | ✅              | `lesson_duration_reached`, `duration_min=45`, количество событий          |
| Какой % занятий используют доску?                  | ✅              | `board_used_meaningfully` / `lesson_duration_reached` (`duration_min=15`) |
| Сколько студентов в среднем на групповом уроке?    | ✅              | среднее `students_count` при `lesson_type=group`                          |
| Сколько занятий не состоялось из-за ошибки токена? | ✅              | `call_connection_failed`, `reason=token_error`                            |
| Конкретная причина ошибки токена (401 vs 500)?     | ❌              | нет — нужны логи `conference-service`                                     |
| Как долго студент пробыл в конкретном занятии?     | ❌              | нет `classroom_id` в событии и нет событий от студента                    |
| Сколько занятий упало из-за разрыва LiveKit?       | ❌              | не трекается — нужно добавить                                             |

### `student_invited_success`

**Когда:** после успешного создания приглашения (POST `/invites/`).
**Зачем:** маркер активации репетитора — он дошёл до шага «позвать ученика».

| Поле          | Тип    | Значения                                                            |
| ------------- | ------ | ------------------------------------------------------------------- |
| `role`        | string | `tutor` / `student` / `parent` / `unknown`                          |
| `source`      | string | `main` / `schedule` / `classroom` / `classrooms` / `invite` / `...` |
| `invite_kind` | string | `student` / `group` / `unknown`                                     |

**Для аналитика:**

- По `source` можно понять, откуда репетиторы чаще всего приглашают учеников — это точка наибольшей конверсии в «начало работы».
- `invite_kind=group` — репетитор работает с группой, важно для сегментации.
- Один репетитор может отправить несколько приглашений одному ученику (пересоздание). Для уникальных репетиторов считайте по `визиту` (session), а не по количеству событий.
- Не означает, что ученик принял — для конверсии смотреть `invite_accepted_success`.

---

### `invite_accepted_success`

**Когда:** после успешного принятия приглашения (POST `/invites/$id/accept`).
**Зачем:** маркер активации ученика — он принял предложение репетитора.

| Поле          | Тип    | Значения                        |
| ------------- | ------ | ------------------------------- |
| `role`        | string | `tutor` / `student` / `unknown` |
| `invite_kind` | string | `student` / `group` / `unknown` |

**Для аналитика:**

- Конверсия приглашений: `invite_accepted_success` / `student_invited_success` (по уникальным пользователям за период). Нормальное значение зависит от канала доставки ссылки — всё вне продукта, метрика только принятий.
- `role` здесь — роль принимающего (студент), не отправителя.

---

### `lesson_created_success`

**Когда:** после успешного создания занятия (POST `/lessons/`).
**Зачем:** фиксирует, что репетитор структурировал учебный процесс (расписание).

| Поле              | Тип     | Значения                            |
| ----------------- | ------- | ----------------------------------- |
| `role`            | string  | `tutor` / `unknown`                 |
| `source`          | string  | страница, с которой создано занятие |
| `lesson_type`     | string  | `individual` / `group` / `unknown`  |
| `is_recurring`    | boolean | повторяющееся занятие               |
| `has_description` | boolean | указано ли описание                 |

**Для аналитика:**

- `is_recurring=true` — репетитор планирует регулярную работу, это сильный сигнал удержания.
- `has_description=true` — косвенный маркер вовлечённости: репетитор заботится об оформлении.
- `source` показывает точку входа: `classroom` означает, что занятие создано внутри конкретного кабинета (уже есть ученик), `schedule` — из расписания (возможно, ещё нет ученика).
- Не говорит ничего о том, состоялось ли занятие — для этого смотреть `lesson_started`.

---

### `lesson_started`

**Когда:** сразу после успешного получения call-токена тьютором, когда `startCallBase` завершился без ошибки. **Не совпадает с кликом по кнопке**: если токен не получен (ошибка сервера, сети), событие не отправляется.

**Только роль `tutor`.**

| Поле          | Тип    | Значения                            |
| ------------- | ------ | ----------------------------------- |
| `role`        | string | всегда `tutor`                      |
| `source`      | string | откуда инициирован старт (pathname) |
| `lesson_type` | string | `individual` / `group` / `unknown`  |

**Для аналитика:**

- Разница между `classroom-start-lesson` (UI-событие клика) и `lesson_started` (outcome-событие) показывает конверсию нажатия кнопки в реальный старт. Если gap большой — проблемы с подключением.
- `source` полезен для понимания, откуда тьюторы стартуют чаще: из кабинета, главной или по ссылке (`goto=call` в URL даёт `source=classroom`).
- Один тьютор может иметь несколько `lesson_started` за день (несколько занятий). Нормально.
- Не означает, что студент подключился — смотреть `lesson_joined`.

---

### `lesson_joined`

**Когда:** сразу после успешного получения call-токена студентом. Аналогично `lesson_started` — не при клике, а при реальном успехе.

**Только роль `student` / `parent`.**

| Поле          | Тип    | Значения                           |
| ------------- | ------ | ---------------------------------- |
| `role`        | string | `student` / `parent` / `unknown`   |
| `source`      | string | откуда инициирован вход            |
| `lesson_type` | string | `individual` / `group` / `unknown` |

**Для аналитика:**

- Соотношение `lesson_started` / `lesson_joined` даёт картину: сколько занятий начал тьютор и в каком проценте студент реально присоединился.
- `source=direct` — студент перешёл по прямой ссылке (не из интерфейса). Хороший маркер для внешних ссылок/уведомлений.
- При групповом занятии (`lesson_type=group`) событие летит от каждого студента отдельно — для подсчёта аудитории суммировать за сессию тьютора.

---

### `call_connected`

**Когда:** `connectionState === 'connected'` в LiveKit. Однократно за сессию — защищено флагом `callConnectedSent` в синглтоне `callSessionState`.

**Отправляется для обеих ролей.**

| Поле          | Тип    | Значения                           |
| ------------- | ------ | ---------------------------------- |
| `role`        | string | `tutor` / `student` / `unknown`    |
| `lesson_type` | string | `individual` / `group` / `unknown` |

**Для аналитика:**

- Это «физическое» подтверждение соединения, в отличие от `lesson_started` / `lesson_joined` (получение токена). Между токеном и подключением пользователь проходит PreJoin экран и жмёт «Войти» — это дополнительный шаг.
- Gap между `lesson_started` (тьютор) и `call_connected` (тьютор) — время на PreJoin (настройка камеры/микрофона). Обычно 10–60 секунд.
- Gap между `lesson_joined` (студент) и `call_connected` (студент) — аналогично.
- Если `call_connected` значительно меньше `lesson_started` — часть тьюторов получает токен, открывает PreJoin и уходит, не нажав «Войти».

---

### `call_connection_failed`

**Когда:** при ошибке подключения к звонку. Два источника:

1. **`token_error`** — HTTP-ошибка при запросе токена у `conference-service` (401, 403, 5xx и т.д.). Код **409 у тьютора** в этот счётчик **не попадает**: 409 означает, что комната требует реактивации — система автоматически делает `reactivateCall` и повторный запрос токена. `token_error` фиксируется только если финальная попытка провалилась.

2. **`permission_error`** — `RoomEvent.MediaDevicesError` от LiveKit: браузер отказал в доступе к камере или микрофону уже после входа в комнату.

> **⚠ Ограничение:** `connection_error` и `unknown` объявлены в типах, но не используются. Разрывы соединения внутри комнаты (обрывы LiveKit после успешного подключения) в это событие **не попадают** — только в консоль. Реальный call failure rate выше, чем показывает это событие.

| Поле     | Тип    | Значения                           |
| -------- | ------ | ---------------------------------- |
| `role`   | string | `tutor` / `student` / `unknown`    |
| `reason` | string | `token_error` / `permission_error` |

**Для аналитика:**

- **Call Failure Rate** = `call_connection_failed` / (`call_connected` + `call_connection_failed`). Нормальный уровень — < 5%. При росте выше 10% — нужно смотреть логи `conference-service`.
- `reason=permission_error` — пользователь не дал доступ к камере/микро. Это часто первый запуск в новом браузере. Если растёт — смотреть на UX PreJoin или добавить обучение.
- `reason=token_error` — проблема на стороне backend или сети. Нужно коррелировать с 5xx в логах.
- Разбивка по `role`: `tutor` > `student` — ожидаемо, тьютор всегда инициирует первым и больше взаимодействует с API.

---

### `lesson_duration_reached`

**Когда:** на временны́х порогах **5 / 15 / 30 / 45 минут** с момента подключения к LiveKit room. Проверка — раз в 30 секунд через `setInterval`, плюс сразу при изменении `connectionState`, `activeBoardId`, `isScreenShareEnabled`. Каждый порог за сессию отправляется ровно один раз (защита через `Set` в `callSessionState`).

**Только тьютор.** Один тьютор = одно занятие → каждое событие = одно занятие, достигшее N минут.

| Поле               | Тип     | Значения                                                                                  |
| ------------------ | ------- | ----------------------------------------------------------------------------------------- |
| `role`             | string  | всегда `tutor`                                                                            |
| `duration_min`     | number  | `5` / `15` / `30` / `45`                                                                  |
| `lesson_type`      | string  | `individual` / `group` / `unknown`                                                        |
| `used_board`       | boolean | открывалась ли доска в течение занятия к этому моменту                                    |
| `used_screenshare` | boolean | использовался ли шаринг экрана к этому моменту                                            |
| `students_count`   | number  | кол-во remote participants в комнате в момент порога (фактически студенты, видны тьютору) |

**Для аналитика:**

- **Основная метрика длительности занятий.** Смотреть воронку: сколько занятий достигло 5 → 15 → 30 → 45 мин. Dropout между порогами показывает, где занятия обрываются.
- `used_board` при `duration_min=15` — доля занятий, где доска была открыта в первые 15 минут. Сравнить с `duration_min=45` для понимания, в какой момент начинают работать с доской.
- `students_count=0` при `duration_min=5` — тьютор 5 минут в одиночестве. Аномалия, студент не подключился. Если таких много — проблема с нотификациями или опытом студента.
- `students_count >= 2` + `lesson_type=individual` — нестандартная ситуация, стоит изучить.
- `lesson_type=group` + `students_count` = фактический охват группового урока.
- **Не использовать** для метрик студенческой активности — студенты это событие не отправляют.

---

### `lesson_finished`

**Когда:** при `connectionState === 'disconnected'` если прошло ≥ 5 минут или достигнут 5-минутный порог. Также при размонтировании компонента (закрытие вкладки, навигация прочь). Защищено флагом `lessonFinishedSent`.

**Отправляется для обеих ролей.**

| Поле               | Тип     | Значения                                      |
| ------------------ | ------- | --------------------------------------------- |
| `role`             | string  | `tutor` / `student` / `unknown`               |
| `duration_bucket`  | string  | `5-15` / `15-30` / `30-45` / `45+`            |
| `lesson_type`      | string  | `individual` / `group` / `unknown`            |
| `used_board`       | boolean | открывалась ли доска за всё занятие           |
| `used_screenshare` | boolean | использовался ли шаринг экрана за всё занятие |

**Для аналитика:**

- Финальный снимок занятия. `duration_bucket` точнее, чем `lesson_duration_reached`, — он отражает реальное время, а не только факт достижения порога.
- Для студента это событие есть, даже если тьютора нет (студент попал в комнату, провёл 5+ минут и вышел). Это редкий, но возможный кейс.
- Сравнение `used_board` в `lesson_finished` с `board_used_meaningfully` даёт конверсию «открыл доску» → «реально использовал».
- Если `lesson_finished` значительно меньше `lesson_duration_reached` (duration_min=5) — часть пользователей закрывает страницу без корректного disconnect (убивает вкладку). Умами может не получить событие при принудительном закрытии вкладки.

---

### `board_opened`

**Когда:** при открытии доски во время звонка (после синхронизации состояния с тьютором, то есть когда доска реально загружена).

**Отправляется для обеих ролей.**

| Поле     | Тип    | Значения                        |
| -------- | ------ | ------------------------------- |
| `role`   | string | `tutor` / `student` / `unknown` |
| `source` | string | откуда открыта (`call` / `...`) |

**Для аналитика:**

- Верхняя часть воронки использования доски: занятие → доска открыта → доска использована значимо.
- В групповом занятии `board_opened` может прийти от нескольких участников за одно занятие (каждый видит доску при открытии). Для подсчёта уникальных занятий с доской лучше использовать `used_board=true` в `lesson_finished`.
- Сравнение `board_opened` с `board_used_meaningfully` показывает, сколько людей открыли доску «посмотреть» и не взаимодействовали.

---

### `board_used_meaningfully`

**Когда:** при meaningful-использовании доски — один раз за одно открытие. Триггеры: длительность просмотра (`duration`), объекты на доске (`objects`), совместная работа (`collaboration`).

**Отправляется для обеих ролей.**

| Поле      | Тип    | Значения                                 |
| --------- | ------ | ---------------------------------------- |
| `role`    | string | `tutor` / `student` / `unknown`          |
| `source`  | string | откуда открыта                           |
| `trigger` | string | `duration` / `objects` / `collaboration` |

**Для аналитика:**

- Ключевая метрика ценности доски. `board_used_meaningfully / lesson_duration_reached (15 min)` = Board Usage Rate.
- По `trigger` можно понять, как именно пользователи «используют» доску: `objects` — рисуют/пишут, `collaboration` — взаимодействуют друг с другом, `duration` — просто долго смотрят.
- Рост `trigger=duration` без `objects` — пользователи смотрят на доску, но не взаимодействуют. Возможно, только тьютор работает, студент наблюдает — нормально для индивидуальных занятий.
- `trigger=collaboration` — наиболее ценный сигнал совместной работы.

---

### Механика `callSessionState`

Все флаги и счётчики звонка хранятся в модульном синглтоне `callSessionState`. Он **сбрасывается** в двух случаях:

- `connectionState === 'disconnected'` (пользователь покинул комнату)
- `token` стал `null/undefined` (сброс перед новой сессией)

Каждый новый вход в звонок — чистый лист. Реконнект внутри LiveKit (`reconnecting → connected`) синглтон **не сбрасывает**, поэтому кратковременные разрывы сети не приводят к повторной отправке уже отправленных порогов.

### Метрики (расчёт в Umami / выгрузке)

#### Активность

| Метрика                | Формула                                                                                          | Примечание                                            |
| ---------------------- | ------------------------------------------------------------------------------------------------ | ----------------------------------------------------- |
| **Active Tutor 7d**    | Уникальные пользователи с `lesson_duration_reached`, `duration_min=15` за 7 дней                 | Основной сигнал активного тьютора                     |
| **Active Tutor 30d**   | Уникальные пользователи с `lesson_duration_reached`, `duration_min=15` минимум 2 раза за 30 дней | Регулярная активность                                 |
| **Active Student 30d** | Уникальные пользователи с `lesson_joined` или `lesson_finished` (`role=student`) за 30 дней      | ⚠ `lesson_duration_reached` студентам не отправляется |

#### Активация

| Метрика               | Формула                                                                                                                  | Примечание                          |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------ | ----------------------------------- |
| **Activated Tutor**   | Пользователь с `student_invited_success` + `lesson_created_success` + `lesson_duration_reached` (≥15 мин) — за всё время | Три шага активации тьютора          |
| **Activated Student** | Пользователь с `invite_accepted_success` + `lesson_joined`                                                               | Принял приглашение и зашёл в звонок |

#### Retention

| Метрика              | Формула                                                                              | Примечание                    |
| -------------------- | ------------------------------------------------------------------------------------ | ----------------------------- |
| **Retained Tutor**   | Второе `lesson_duration_reached` (`duration_min=15`) в течение 14 дней после первого | Вернулся и провёл ещё занятие |
| **Retained Student** | `lesson_joined` минимум 2 раза за 30 дней                                            |                               |

#### Качество занятий

| Метрика                     | Формула                                                                   | Примечание                                   |
| --------------------------- | ------------------------------------------------------------------------- | -------------------------------------------- |
| **Lesson completion rate**  | `lesson_duration_reached` (45 мин) / `lesson_duration_reached` (5 мин)    | Доля занятий, дошедших до 45 мин             |
| **Avg lesson duration**     | Распределение `lesson_duration_reached` по `duration_min`                 | Каждое значение = занятие, достигшее порога  |
| **Avg students per lesson** | Среднее `students_count` из `lesson_duration_reached` (`duration_min=15`) | Наполненность занятий                        |
| **Board Usage Rate**        | `board_used_meaningfully` / `lesson_duration_reached` (`duration_min=15`) | Доля занятий с реальным использованием доски |
| **Screenshare Rate**        | `lesson_finished` (`used_screenshare=true`) / `lesson_finished`           | Доля занятий с шарингом экрана               |

| ------------------------- | ------------------------------------------------------------------------ | ---------------------------------------------------------------------- |

| **Token → Connect Rate**  | `call_connected` / `lesson_started` (тьютор)                             | Конверсия получения токена в подключение к комнате                     |

### Подводные камни

1. **`lesson_duration_reached` — только тьюторы.** Не использовать для метрик студентов. Студентов считать через `lesson_joined`, `lesson_finished` (`role=student`).

2. **Один тьютор — одно занятие (в норме).** Если один тьютор в день заходит в звонок несколько раз (отключился и перезашёл), каждый вход — новая сессия, новые пороги.

3. **`lesson_finished` может не долететь** при жёстком закрытии вкладки (kill process, потеря питания). Браузер не успевает отправить beacon. При анализе длительности лучше опираться на `lesson_duration_reached`.

4. **`students_count` — snapshot, не среднее.** Это количество участников в момент достижения порога. Если студент зашёл позже 15-минутной отметки, `students_count` при `duration_min=15` будет меньше реального.

5. **`call_connection_failed` не покрывает разрывы после подключения.** Если пользователь вошёл в комнату, а потом LiveKit потерял соединение — это не считается в `call_connection_failed`. Реальная надёжность ВКС требует анализа логов LiveKit.

6. **`board_opened` дублируется в групповых занятиях** — летит от каждого участника, открывшего доску. Для уникальных занятий с доской использовать `used_board=true` в `lesson_finished` или `lesson_duration_reached`.
